# Coding Norms — Rationale and Detail

This document explains the reasoning behind each norm in `AGENTS.md`. The norms are opinionated and optimized for a specific workflow: one human designer/product owner, AI agents doing implementation via GLM/OpenCode, incremental work orders.

## Memory is the DB

callmem captures every event and extracts entities into `.callmem/memory.db`. The daemon rewrites `SESSION_SUMMARY.md` on each extraction pass, and `mem_get_briefing` serves the same data via MCP.

The old flat memory files (`.callmem/SESSION.md`, `TODO.md`, `DECISIONS.md`) were retired once the daemon proved reliable. The DB is the source of truth; the briefing is a rendering of it.

### Why the DB over flat files

- The daemon keeps it in sync automatically — no "update memory at session end" discipline required
- Entities are structured and queryable (by type, status, date, file, session) rather than free-form markdown
- Session boundaries, event lineage, and file attribution are preserved
- Briefings can be compressed, focused, and token-budgeted on demand

### What still takes a manual `mem_ingest`

The extractor is good at decisions/todos/bugfixes inside conversation, but it can't see the intent behind what you chose *not* to do. If you want something remembered verbatim — an architectural rationale, a gotcha, a priority ordering — call `mem_ingest` explicitly with the right type.

## Git Discipline

### Why commit often

Sessions can crash, timeout, or hit rate limits. Uncommitted work is lost. Small, frequent commits create natural checkpoints and give the extractor a clean per-commit boundary to attribute work against.

### Why push immediately

Same reason: pushed commits survive session loss. Additionally, if you're working from multiple machines (desktop + iPad over SSH), pushing keeps everything in sync.

### Why conventional commits

They're parseable, and callmem attributes extracted entities to commits where possible. They also make git log readable at a glance:

```
feat(WO-04): implement event ingest pipeline
test(WO-04): add FTS5 population verification
fix: handle dedup window edge case with identical timestamps
docs: clarify retention policy in coding-norms
```

### Why no branches for now

With one developer and sequential work orders, branches add overhead without benefit. Merge conflicts with yourself are pointless friction. If a WO is genuinely risky, a branch provides a rollback point — but that's the exception.

## Code Quality

### Why no AI attribution

1. It's noise — adds no information for the next reader (human or AI)
2. It dates poorly — "Generated by Claude 3.5" tells you nothing useful
3. It can leak workflow details you don't want in public repos
4. It creates a two-tier codebase feeling — "AI code" vs "real code" — which discourages ownership

### Why type hints everywhere

GLM and other coding agents are significantly better at working with typed code. Type hints serve as documentation, enable mypy static analysis, and reduce the chance of the agent generating incompatible interfaces between modules.

### Why the repository pattern

When an AI agent writes SQL, it tends to scatter queries across the codebase, use slightly different patterns in each location, and occasionally generate unsafe string formatting. Centralizing all SQL in `repository.py`:
- Makes SQL reviewable in one place
- Enforces parameterized queries by convention
- Keeps the engine testable with a mock repository
- Prevents the agent from inventing new query patterns in every file

### Why strict on error handling

AI agents have a tendency to write broad `except Exception` blocks or swallow errors silently. This hides bugs during development. The norm is: fail loudly now, add graceful degradation later when the failure mode is understood.

## Session Workflow

### Why "read before write"

The single most common failure mode with coding agents is: the agent doesn't read existing code before writing new code, and produces something incompatible with what exists. The AGENTS.md instruction to read `SESSION_SUMMARY.md` / call `mem_get_briefing` first, and read target files before modifying them, directly addresses this.

## Anti-patterns to Watch For

Things that GLM (or any coding agent) tends to do that the norms are designed to prevent:

| Anti-pattern | Norm that prevents it |
|---|---|
| Rewriting working code unnecessarily | "Read existing code before writing new code" |
| Giant uncommitted changes | "Commit after each meaningful step" |
| Ignoring failing tests | "Fix before moving on" |
| Bare `except:` blocks | "Catch specific exceptions" |
| SQL scattered across files | "SQL lives in repository.py" |
| Vague TODO comments | "Say what and why" |
| Starting the next WO before finishing the current one | "Complete all acceptance criteria before moving on" |
| `# Generated by AI` in code | "Never include AI attribution" |
| Inventing new patterns instead of following existing ones | "Match the style of adjacent modules" |
| Not pushing before ending a session | "Never leave with unpushed commits" |
